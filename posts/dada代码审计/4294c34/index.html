sql注⼊的起源与发展:

SQL注⼊第⼀次为公众所知，是在1998年的著名⿊客杂志《Phrack》第54期上，⼀位名叫rfp 的⿊客发表了⼀篇题为“NT Web Technology Vulnerabilities”的⽂章。 在⽂章中，第⼀次向公众介绍了这种新型攻击。下⾯是当时给出的例⼦，现在看已经很古⽼ 了。是伪代码的表述形式。

 ShipCity = Request.from("ShipCity");
 var sql = "select * from OrderTable where ShipCity = '" + ShipCity + 
"'";

从那开始乃⾄⼏年前(2015,2016)，sql注⼊都⼀直名列⽹站安全、web安全的头把交椅，主要就是因为其 攻击简单、攻击⼿法多样、危害巨⼤（拿数据库、什么拿shell、拿系统权限）。 近⼏年，因为规范化的框架使⽤、orm的普及，基本在框架层⾯就解决了⼤部分普通sql注⼊ 的问题，sql注⼊越来越难，所以表现就是sql注⼊不那么受关注了。但需要注意的是，sql注⼊ 仍然是危害巨⼤的攻击⼿法。

为什么错误

在 MyBatis 中，SELECT * FROM table WHERE username = '${value}' 和 SELECT * FROM table WHERE username = '#{value}' 的主要区别在于它们的参数替换方式以及可能带来的 SQL 注入风险。

${}

  • 原理：${} 是 MyBatis 的字符串替换占位符，它会在 SQL 语句生成前直接替换成实际的参数值。
  • 风险：因为 ${} 是直接替换，所以如果参数 value 包含了恶意的 SQL 代码，就会导致 SQL 注入漏洞。例如，如果 value 是 "admin' OR '1'='1"，生成的 SQL 语句会变成 SELECT * FROM table WHERE username = 'admin' OR '1'='1'，这会导致数据库返回所有用户的数据。
  • 使用场景：适用于动态表名或列名等情况下，但不建议用于用户输入的参数值

#{}

  • 原理：#{} 是 MyBatis 的参数占位符，它会在 SQL 语句生成时使用预编译的方式将参数值传递给数据库。( 相当于出去取字符串)
  • 安全性：因为参数值是通过预编译传递的，数据库会将其作为数据处理，而不会执行其中的 SQL 代码，从而避免 SQL 注入漏洞。例如，如果 value 是 "admin' OR '1'='1"，生成的 SQL 语句会变成 SELECT * FROM table WHERE username = ?，并且参数值 admin' OR '1'='1' 会作为数据传递给数据库。
  • 使用场景：建议始终使用 #{} 来传递用户输入的参数，以确保 SQL 注入安全性。

简单讲就是MyBatis有两种变量绑定⽅式，分别是：#{}和${}

 #是绑定变量的形式，底层会⽤#{}会被替换为?号，有参数映射,会在
DefaultParameterHandler中进⾏设置占位符的操作  -->预编译
$也是绑定变量的形式，{value}是直接被替换为了对应的值，没有参数映射，不会进⾏设置占位
符的操作   -->拼接

所以这也是很多代码审计初学者或者很多课程，上来就让⼤家去找$符号的原因。⼤家后续遇 到MyBatis也确实可以这么搞，对于⼩cms简单有效。

核⼼思维 

sql注⼊，就是在执⾏⼀段sql语句，关键是数据库类型

核⼼思维就是：想法设法去执⾏⼀条完全的sql语句，把数据带出来或把命令传进去。 与之将对的，以下虽然是问题的⼀部分，但不能称之为本技术的核⼼思维。

编程语⾔

不同编程语⾔最终的⽬的都是为了将我们的payload送⼊数据库层进⾏执⾏，能看到注⼊点即 可，语⾔不重要。

注⼊类型

不需要特别关注分类，所有的分类都是sql语句的不同写法⽽已，⽐编程语⾔相对重要⼀些。

  1. 能否堆叠注⼊

    我们之前说了，⼀条sql语句，最终会被解析成⼀段控制序列

     action(动作): select
 object（对象）: table
 subject（⽬标客体）: *
 condition（条件）:
 key: username
 value: $username // ⽤户输⼊

    如果能够使⽤堆叠注⼊，我们就能跳出action动作，执⾏其他动作（⼀条新的sql语句），肯定 危害更⼤。 ⽬前来看，堆叠注⼊越来越少。经验上来看，mssql数据库的堆叠注⼊最多；oracle数量基本 差不多，稍微少⼀点；mysql不多。

  2. 爆数据的速度上

    报错~=联合查询>带外数据>布尔盲注>时间盲注

输⼊点⽐上⾯相对⼜重要⼀些。因为输⼊点决定了我们能⽤什么样的Vector（攻击向量），以 及是否需要绕过。这个下⾯会结合绕过去讲。

宽字节注⼊

在utf-8还不那么通⽤的时候，各种⽜⻤蛇神都群魔乱舞。各种字符集都存在。（包括现在， 如果你去搞⼀个jp、或者kr的站，很可能碰到其他的宽字符集）就造成了⼀种新的注⼊⽅式， 叫宽字节注⼊。

概念 

我们知道字节是计算机存储世界中最⼩的衡量单位，1Byte = 8bits。所以⼀个字节最⼤能够表
示2^8=256个字符。
所以：
  对ascii编码⽽⾔，⼀个字符⽤⼀个字节就可以表示，所以ascii编码最多可以表示256个字
符。
  对GBK编码⽽⾔，⼀个汉字字符需要⽤两个字节表示。所以gbk编码理论上最多可以表示
256*256个字符

关于字符集、字符编码的更多知识，⼤家可以去百度了解。

利⽤

因为宽字符的存在，导致⼀些防注⼊的⽅式会被绕过。我们以代码的形式来讲解。

<?php
 $db = init_db();
 $db->query("set SET NAMES 'gbk');  //设置gbk字符集
$username = addslashes($_GET['username']);  //input: fuckdada' and 1=1#
 $db->query("select * from table where username = '$username'");  
?>

我们来考虑下sql语句会是怎么样的呈现,输⼊fuckdada’ and 1=1#

在这种场景下，我们没办法进⾏sql注⼊，因为我们的单引号被转义了，我们没办法侵⼊到控 制流去。 输⼊fuckdada%df’ and 1=1# 数据在存储的时候⼀定是以字节存储的，但是数据解读的时候，都是以字符的标准去解读的。 所以，在连接数据库进⾏sql执⾏（执⾏就是⼀种对存储的解读）时，会按照字符集编码的规 范去解读，所以遇到了\xDF\x5C的时候，会解读成⼀个字符。

此时我们来关注下输⼊流转

url输⼊-->php字符串变量-->addslashes-->sql语句-->数据库
fuckdada%DF' and 1=1#  --> fuckdada\xDF' and 1=1# --> fuckdada\xDF\' and 
1=1# -->
select * from table where username = 'fuckdada運' and 1=1#'  //数据流成功⼊侵到控制流

有个在线⽹站 ⼤家可以试试编码转换 这⾥有个点要讲解⼀下，我在写这个⽂档翻阅资料的时候，发现⽹络上⼤部分宽字符注⼊，都 是以%df作为逃逸字符来进⾏逃逸。但是没有讲原因。原因在这⾥ 其实并不必须⽤%df，符合要求的编码即可。

所以，代码审计中审阅宽字节注⼊的⽅式，就是查看数据库连接⽂件（⼀般名字类似 conn.php），检查其字符集类型是什么，如果不是utf-8（因为脚本的字符集默认是utf-8）就 可能有注⼊产⽣。 另外给⼤家贴⼀点参考资料

Hibernate

Hibernate是⼀种ORM框架，⽤来映射与tables相关的类定义（代码），并包含⼀些⾼级特 性，包括缓存以及继承，通常在Java与.NET中使⽤（可参考NHibernate），但在Java⽣态系 统中更受欢迎。近来似乎逐渐有被Mybatis取代的趋势。

like关键字

我们知道sql语句的模糊查找⾥⾯⽤的关键字like，⽽like关键字默认是不会预编译的(如果使⽤ Mybatis则是预编译报错)。数据库⽅给出的原因好像是like预编译会造成慢查询和DOS。

只能⼿动去添加预编译

<?php
 $username = $_GET['username'];
 $db = "mysql:host=127.0.0.1;dbname=test;";
 $dbname = "root";
 $passwd = "root";
 $conn = new PDO($dbs, $dbname, $passwd);
 $conn->query('SET NAMES GBK');
 $stmt = $conn->prepare("select * from table where username like 
'%:username%'");  //不⽣效
$stmt = $conn->prepare("select * from table where username like 
concat('%',:username,'%'");  //⽣效
$stmt->bindParam(":username",$username);
 $stmt->execute();
 ?>

或者⼀些java的开发，Mybatis编译报错，然后他们⾃⼰去添加过滤（过滤没写好）或者不过 滤（使⽤原⽣语句），导致GG。

与之类似的还有IN关键字，该位置也默认不能预编译，需要在预编译语法中去for循环，有些 程序员为了⽅便，可能也会在这边偷⼯减料。

不能加引号的关键字

我们刚才分析了预编译模式下的宽字节注⼊，我们可以发现，预编译+绑定变量的效果，有点 类似于做了两个步骤

$newInput = addslashes($input)   //内容转义
sql = select * from table where column = '$newInput'  //强制⽤单引号包裹

那么，结合我们刚才讲的，产⽣注⼊的输⼊点

select $username$,password from $table$ where $username2$ = '$dada$' 
order by $username3$ $desc$ limit $0$,1

是否有输⼊点是必然不能加单引号的呢，如果不能加单引号，那么就不能预编译 于是我们找到了

$username$,$username2$,$table$,$username3$,$desc$,$0$

这些地⽅都是不能加单引号的，总结就是

表名、列名、limit⼦句、order by[desc/asc]

权限的基本关系

权限的基本关系，其实没啥好讲的，就⼀句话：

认清数据平⾯

eg:

在cms/web层⾯：web后台管理员权限>web前台会员权限>游客权限
在数据库层⾯:root⽤户/sa⽤户/postgres⽤户>普通杂七杂⼋⽤户
在操作系统层⾯:root > 所有其他⽤户

这还⽤讲吗？

在同⼀个数据平⾯上，权限关系很好理清。 关键还是这句话：

未授权rce

如果我们以游客的权限，拿到⼀个rce漏洞（也就是未授权rce），那么我们的权限是什么？是 web后台管理员吗？⾮也。此时我们的数据平⾯是在操作系统层⾯。

在操作系统层⾯，我们的bash进程是httpd进程派⽣的，所以与httpd进程是同层级的，所以此 时我可以读写web⽬录的任意⽂件（前提是web⽬录本身对httpd可读写）

在数据库层⾯，我们⽬前就是mysql进程维护的⼀个通信线程，我们就是数据库的root或者普 通库权限（取决于web数据库配置⽂件中定义的⽤户）。

如果我们以游客权限，拿到⼀个sql注⼊点，并且这个注⼊点可以执⾏命令，那么我们的权限 是什么？是数据库的root吗？⾮也。此时我们的数据平⾯是在操作系统层⾯。

在操作系统层⾯，我们⽬前就是mysql进程派⽣的⼀个bash进程，所以与mysql进程是同层级 的，所以此时我可以读写mysql数据库⽂件⽬录的⽂件。也⼤概率有权限读web⽬录的⽂件
